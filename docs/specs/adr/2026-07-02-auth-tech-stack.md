# ADR: Технологический стек auth-подсистемы

Дата: 2026-07-02
Статус: Принято

## Контекст

Для реализации auth-подсистемы (Steam OpenID, JWT/refresh токены, SQLite-хранилище сессий) нужно выбрать конкретные библиотеки. Проект распространяется как один Go-бинарник без внешней инфраструктуры (см. README) — это ограничение напрямую влияет на выбор.

## Решения

### 1. JWT (RS256): `github.com/golang-jwt/jwt/v5`

**Альтернативы:** `gbrlsnchs/jwt`, `kataras/jwt`, `hashicorp/cap/jwt`.

| Критерий | golang-jwt/jwt/v5 | Альтернативы |
|---|---|---|
| Актуальность | Релиз январь 2026, активно поддерживается | Разная степень поддержки, меньше community |
| Adoption | ~15 000+ проектов-импортёров | На порядок меньше у большинства |
| RS256 | Нативная поддержка, документированный паттерн | Есть, но меньше проверенных примеров в проде |
| Статус | Официально «production ready», используется как преемник dgrijalva/jwt-go | — |

**Решение:** `golang-jwt/jwt/v5`. Де-факто стандарт в Go-экосистеме для JWT, наибольшая проверка в проде, прямая поддержка RS256.

### 2. SQLite-драйвер: `modernc.org/sqlite`

**Альтернатива:** `mattn/go-sqlite3` (CGO-обёртка над оригинальной C-библиотекой).

| Критерий | modernc.org/sqlite (pure Go) | mattn/go-sqlite3 (CGO) |
|---|---|---|
| Кросс-компиляция | Тривиальна, `CGO_ENABLED=0`, статический бинарник | Требует C-тулчейн под каждую целевую платформу |
| Соответствие цели «один бинарник» | Прямое соответствие | Усложняет сборку и дистрибуцию |
| Производительность | Ниже (по бенчмаркам — до ~1.5-2x медленнее на некоторых сценариях) | Выше, использует нативный код |
| Зрелость | Активно поддерживается (последний релиз — июнь 2026), используется рядом проектов именно ради отказа от CGO (например, миграция gogs) | Де-факто стандарт, дольше в проде |

**Решение:** `modernc.org/sqlite`. Нагрузка на БД в этом проекте — сессии и небольшой конфиг, не производительность bottleneck. Простота сборки/дистрибуции одного бинарника перевешивает разницу в скорости.

**Риск:** если позже потребуется более тяжёлая работа с БД (например, метаданные демок в больших объёмах), стоит переоценить это решение — modernc.org/sqlite может стать узким местом. Зафиксировать как пункт для ревизии при добавлении parser/demo-storage подсистемы.

### 3. HTTP-роутинг: стандартная библиотека `net/http` (Go 1.22+ ServeMux)

**Альтернативы:** `chi`, `gorilla/mux`, `gin`.

Начиная с Go 1.22, `http.ServeMux` поддерживает method-based и pattern-based роутинг (`mux.HandleFunc("POST /auth/refresh", ...)`), что закрывает основную причину исторически тянуть `chi`/`gorilla`. Для auth-подсистемы (4 маршрута) сторонний роутер не добавляет ценности, только лишнюю зависимость.

**Решение:** стандартная библиотека. Пересмотреть, если позже понадобится сложная маршрутизация (route groups, встроенные middleware chains) — тогда `chi` — разумный кандидат, он совместим с `net/http` без переписывания хендлеров.

### 4. Steam-аутентификация: ручная реализация OpenID 2.0 вместо готовой библиотеки

**Альтернатива:** `yohcop/openid-go` (используется в найденных Steam-примерах вроде `pneisen/steam-openid-go`).

| Критерий | Ручная реализация | yohcop/openid-go |
|---|---|---|
| Поддержка | Наш код, полностью под контролем | Последняя активность — годы назад, репозиторий фактически заброшен |
| Объём | ~80 строк (redirect URL + check_authentication запрос) | Полноценная generic OpenID 1.1/2.0 discovery-библиотека — избыточна для единственного фиксированного provider |
| Поверхность атаки | Минимальна, весь путь — 2 HTTP-запроса к фиксированному Steam endpoint | Больше кода = больше поверхность, включая неиспользуемый discovery-механизм |
| Гибкость под будущее | Полный контроль над edge cases (см. Task 7 плана) | Нужно подстраиваться под API библиотеки |

**Решение:** ручная реализация. Steam — единственный и не меняющийся identity provider с фиксированным endpoint (`steamcommunity.com/openid`), поэтому generic OpenID discovery не нужен — только encode redirect URL и `check_authentication` verification, что задокументировано в открытой Steamworks-документации.

## Последствия

- Зависимости `go.mod`: `golang-jwt/jwt/v5`, `modernc.org/sqlite`. Никаких дополнительных HTTP- или OpenID-библиотек.
- Сборка не требует C-тулчейна — `CGO_ENABLED=0` совместимо с целью «единый бинарник без внешней инфраструктуры» из README.
- Пункт на будущее ревью: производительность `modernc.org/sqlite` при добавлении demo-storage подсистемы (см. риск выше).

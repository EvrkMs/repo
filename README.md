# Intact-CS-Map

Intact-CS-Map — это специализированное решение для тактического разбора демок CS2 в режиме реального времени. Проект создан для тренерских штабов и команд, которым необходимо интерактивно разбирать игровые ситуации на «большом экране» в условиях буткемпа или тренировочной базы без лишних задержек.

## Основная концепция

Инструмент предоставляет веб-интерфейс для анализа демок CS2. Тренер загружает демку, получает наглядную 2D-карту с перемещениями игроков и событиями раунда. Главная особенность — возможность «расшарить» сессию на второе устройство (ТВ, проектор) по QR-коду: все графические пометки и перемещения по таймлайну, сделанные тренером, мгновенно отображаются на втором устройстве в реальном времени.

## Почему это нужно?

- **Локальная работа:** все данные хранятся на вашем сервере, тактики не покидают пределы вашей сети.
- **Sync:** никаких «демонстраций экрана» через Discord с задержками. Мы передаем только координаты рисования и метаданные.
- **Автономность:** инструмент поставляется в виде одного бинарного файла, который не требует сложной инфраструктуры или внешних баз данных.

## Конфигурация (Environment)

Приложение настраивается через файл `.env`. Доступ к системе строго ограничен закрытым списком SteamID — этот список **не редактируется через веб-интерфейс** ни при каких обстоятельствах (см. раздел «Модель доступа» ниже).

Пример `.env`:

```dotenv
# Базовые настройки
PORT=8080
DB_PATH=./data/intact.db
DEMO_STORAGE_PATH=./data/demos

# Steam API
STEAM_API_KEY=your_steam_api_key_here

# Root-администратор: break-glass доступ, не хранится в БД,
# доступен всегда независимо от состояния SQLite
ROOT_STEAM_ID=76561198000000001

# Закрытый список допущенных SteamID (через запятую).
# Единственный источник допуска в систему. Изменяется
# только правкой этого файла + рестартом сервера.
STEAM_ADMIN_IDS=76561198000000002,76561198000000003

# TLS (опционально)
TLS_CERT_PATH=
TLS_KEY_PATH=

# JWT (RS256, асимметричная подпись). Если файлов нет —
# генерируются автоматически при первом старте сервера.
JWT_PRIVATE_KEY_PATH=./data/keys/jwt_private.pem
JWT_PUBLIC_KEY_PATH=./data/keys/jwt_public.pem

# Секреты
SESSION_SECRET=your_super_secret_key
```

## Модель доступа

Список допущенных SteamID — **закрытое множество**, полностью определяемое `.env`:

```
is_admin(steam_id) := steam_id == ROOT_STEAM_ID
                    OR steam_id ∈ STEAM_ADMIN_IDS
```

- SQLite **не хранит** список допущенных SteamID и не участвует в проверке доступа.
- Панель управления **не может** добавить нового человека в список допуска — такой функциональности не существует ни для root, ни для кого-либо ещё.
- Расширение круга людей с доступом — намеренно "тяжёлая" операция: правка `.env` + рестарт бинарника. Это защита на случай компрометации веб-слоя (auth bypass, IDOR и т.п. в панели) — атакующий не может выдать себе доступ через уязвимость в приложении, поскольку механизма выдачи доступа через веб физически нет.

### Что тогда делает панель управления

Панель root'а управляет **рантайм-конфигом сервера**, а не списком доступа:

- Разрешённый IP reverse-proxy (см. «TLS и Secure cookie» ниже)
- Прочие серверные настройки (лимиты, таймауты сессий и т.п.)

Эти настройки хранятся в SQLite и меняются без рестарта бинарника.

## Аутентификация

- **OAuth2:** аутентификация проходит через официальную систему Steam OpenID.
- **Access control:** сервер проверяет SteamID пользователя при **каждом** запросе к защищённым методам по правилу из раздела «Модель доступа» — не только при логине/refresh. Если ID не входит в допущенное множество, доступ блокируется немедленно, даже если у клиента есть формально ещё не истёкший access/refresh токен. Благодаря этому отдельная логика принудительной инвалидации сессий при удалении SteamID из `.env` не нужна: `is_admin` отсекает доступ раньше, чем дело доходит до проверки токена по существу.
- **Токены:** гибридная схема.
  - Access token — JWT, подпись **RS256** (асимметричная, RSA-2048), короткоживущий (10–15 мин). Приватный ключ используется только на сервере, публичный можно безопасно передать другому сервису при будущем расширении системы — этим RS256 выгоднее HS256, где секрет пришлось бы шарить.
  - Refresh token — хранится в SQLite **только в виде хэша (SHA-256)**, живёт 1 день, с rotation и reuse detection (при повторном использовании уже инвалидированного refresh-токена вся цепочка сессии инвалидируется и событие логируется). Хэширование ограничивает ущерб при краже файла `intact.db` — открытый токен из БД не восстановить.
  - Просроченные/инвалидированные refresh-токены — просто удаляются (периодическая очистка по `expires_at < now()`), без дополнительной логики revoke.
- **Cookie:** refresh-токен передаётся через `httpOnly` cookie.
  - `SameSite=Strict`, если `Secure=true`; иначе `SameSite=Lax`.
- **CSRF:** отдельный CSRF-токен не используется. Приложение работает на одном домене, кросс-сайтовые запросы к API не предполагаются, и `SameSite` уже блокирует отправку cookie при кросс-сайтовых запросах. Единственное условие, при котором это остаётся безопасным: **все state-changing операции (загрузка демки, изменение конфига, revoke и т.п.) выполняются только через `POST`/`PUT`/`DELETE`, никогда через `GET`** — при `SameSite=Lax` (fallback для голого HTTP) cookie всё ещё уходит при top-level GET-навигации с другого сайта.

## TLS и Secure cookie

`Secure` определяется автоматически, вручную не переключается:

```
если RemoteAddr запроса ∈ allowed_proxy_ip (SQLite, настраивается в панели)
    → доверяем заголовку X-Forwarded-Proto от этого запроса
иначе
    → Secure = (r.TLS != nil)  // факт реального TLS-хендшейка на самом сервере
```

Поддерживаемые сценарии развёртывания:

1. **Прямой TLS в бинарнике** — указать `TLS_CERT_PATH`/`TLS_KEY_PATH` в `.env`. Сервер сам терминирует TLS, `Secure=true` включается автоматически по факту хендшейка.
2. **Голый HTTP в LAN** (без сертификата и без прокси) — `Secure=false`, `SameSite=Lax`.
3. **За reverse-proxy** — прокси должен обеспечивать реальный TLS до бэкенда:
   - **Double TLS** — прокси терминирует TLS от клиента и заново шифрует соединение к бэкенду (`proxy_pass https://...` в nginx, `transport https` в Caddy). Бэкенд видит настоящий `r.TLS != nil`.
   - **L4/stream passthrough** — прокси не расшифровывает трафик, TLS-хендшейк идёт напрямую клиент↔бэкенд (SNI-based routing). Бэкенд также видит настоящий `r.TLS != nil`.
   - Если по инфраструктурным причинам TLS терминируется только на прокси, а к бэкенду идёт голый HTTP — нужно добавить IP прокси в `allowed_proxy_ip` через панель, тогда `Secure` определяется по `X-Forwarded-Proto`, но только для запросов с этого доверенного IP.

Если `allowed_proxy_ip` настроен, сервер должен слушать на внутреннем интерфейсе (например `127.0.0.1:8080`), а не `0.0.0.0`, чтобы исключить обход прокси и подделку заголовка напрямую.

## Безопасность

- **OAuth2:** аутентификация проходит через официальную систему Steam OpenID.
- **Access control:** закрытый список SteamID в `.env`, без возможности расширения через веб-интерфейс (см. «Модель доступа»).
- **Rate limiting:** upload-эндпоинт ограничен per-session, чтобы исключить флуд демками.
- **Локальное хранение:** использование SQLite обеспечивает безопасность — никакие облачные провайдеры не имеют доступа к вашим файлам.
